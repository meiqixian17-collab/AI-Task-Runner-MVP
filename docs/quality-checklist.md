# 质量打磨检查清单

这份清单用于后续每轮迭代前后检查，避免项目从“任务执行系统”退回普通 AI 生成器。

## P0 核心闭环

- 用户输入任务后，系统能生成一个短、具体、可立刻开始的当前步骤。
- 当前步骤不能只是“整理思路”“准备一下”“制定计划”这类抽象动作。
- 用户完成当前步骤后，下一步必须自然承接 `stepHistory`，不能重启任务。
- 简单任务完成 2 步或遇到重复步骤时，应进入完成确认，而不是继续强行生成下一步。
- AI 认为任务完成时，能进入 completed 状态，而不是无限生成下一步。
- AI 失败、超时或返回异常时，前端仍能给出 fallback step。

## 澄清流程

- 信息不足时，系统先问一个必要问题，而不是直接猜。
- 澄清问题只收集文本信息，不要求用户上传图片、文件或音频。
- 用户补充信息后，系统能继续推进原任务。
- 澄清不应阻塞所有任务；信息足够时应直接生成行动步骤。

## 卡点恢复

- 情绪/社交压力：先移除发送、联系、被评价压力，优先安全草稿。
- 认知模糊：先建立判断标准或输出边界。
- 完美主义：先降低质量门槛，例如 60 分版本或粗糙草稿。
- 身体低能量：允许暂停并保留恢复入口，不强推站起、出门或运动。
- 价值怀疑：先判断是否值得继续，而不是硬推原任务。
- fallback step 必须保留原任务目标，不能变成无关建议。

## UI / UX

- 每个状态下只有一个最明显的主操作。
- idle 状态突出任务输入。
- ready / executing 状态突出 StepCard。
- 历史记录、辅助信息和调试信息不抢主行动。
- 移动端按钮可点区域足够大，文本不溢出。
- 错误信息靠近相关操作，并给出恢复路径。

## 工程质量

- `npm run test:quality` 通过。
- `npm run test:resistance` 通过。
- `npm run build` 通过。
- README 与展示文档能让新人独立启动和演示项目。
- `.env` 不进入 Git，真实 API key 不写入文档或源码。
- 新增功能必须至少补一个对应质量样例或回归测试。
