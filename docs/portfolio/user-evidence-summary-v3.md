### 11.9 用户问题证据与试用反馈摘要

> 说明：这一节作为支撑证据使用，访谈样本参考原文档「10 用户问题证据：访谈样本」。当前试用反馈仍是草稿，正式投递前建议替换为真实试用对象、真实任务过程和用户原话。

**证据结论：** 3 位目标用户反复出现的不是“不会做任务”，而是启动、推进和恢复三个断点。这个证据支持我把 AI Task Runner 的核心设计放在“当前一步”上，而不是完整计划上。

| 证据发现 | 代表引用 | 对应设计决策 |
|---|---|---|
| 用户知道目标，但不知道第一步 | “我知道我要投实习，也知道简历要改，但一打开项目经历那一栏，就不知道第一句该写什么。” | StepCard 每次只呈现一个当前可执行动作 |
| 完整计划可能增加压力 | “计划列出来以后，我会觉得自己又多了一堆没做的事。” | 不一次性展示完整计划，先降低启动成本 |
| 卡住常来自评价压力和完美主义 | “我总觉得自己写得很学生气，拿不出手。” | 提供 60 分版本、粗糙草稿、只写一句等低门槛入口 |
| 完成一步后仍可能断掉 | “查完以后不知道接下来是整理观点，还是先写正文。” | 基于 stepHistory 生成自然下一步 |
| 中断后重新进入困难 | “隔一天再回来，我已经忘了之前卡在哪，又要重新想一遍。” | 保留 re-entry point，让用户从上次位置继续 |
| 社交或提交任务会带来暴露压力 | “有些消息不是很难回，但我会怕对方追问。” | 先写不发送的安全草稿，而不是直接推动提交 |

**试用反馈草稿（待核实）**

| 用户 | 真实任务 | 原型介入方式 | 使用后反馈 |
|---|---|---|---|
| U01 | 修改简历项目经历，准备投递实习 | 先生成“60 分版本的项目经历首句”，再引导只写一句、不修改 | “比起直接让我优化整份简历，先写一句更容易开始。” |
| U02 | 推进期末课题报告 | 先问交付物和截止时间，再生成“找 3 条资料并各写一句观点” | “它没有一上来列很多计划，而是告诉我现在先做什么。” |
| U03 | 处理拖延的学习/沟通任务 | 保留 re-entry point，并先写不发送的安全草稿 | “先写不发送的草稿会比较安心。” |
