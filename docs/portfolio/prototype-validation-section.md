## 09 原型验证与体验边界

我做可运行原型的目的不是展示工程能力，而是让核心交互闭环可以被点击、观察和验证。这个原型服务于产品/交互判断：用户能否更容易开始，是否理解当前一步，卡住时是否能被带回行动。

| 验证问题 | 设计判断 | 原型中的体现 |
|---|---|---|
| 用户能否从模糊任务进入行动 | 入口不要求用户先拆计划，而是让系统生成当前一步 | 用户输入任务后直接得到 StepCard |
| AI 输出是否能变成可执行状态 | 不把建议停留在聊天文本里，而是包装成可开始、可完成、可恢复的执行单元 | StepCard 承载当前动作、执行状态和操作入口 |
| 信息不足时是否应该继续生成 | 与其猜测，不如只问当前步骤必要问题 | 澄清问题只补最小上下文 |
| 用户卡住时如何恢复 | 不继续催促，而是根据阻力类型降低门槛 | fallback step、暂停与恢复入口 |
| 任务推进是否能保持连续 | 完成一步后需要自然衔接，而不是让用户重新判断下一步 | 用已完成步骤作为下一步生成依据 |

因此，原型在作品集中承担的是“验证设计机制”的角色，而不是展示技术实现复杂度。技术实现只作为支撑手段，不作为本项目的主要卖点。
