# AI Task Runner 展示脚本

这份脚本用于作品集、答辩或项目演示。建议演示时打开前端页面、终端测试结果和项目结构，让观众同时看到产品体验和工程质量。

## 一句话定位

AI Task Runner 是一个把“我不知道怎么开始”转成“现在只做这一步”的任务推进系统。

它的重点不是生成完整计划，而是在任务执行过程中持续处理三件事：

- 当前最小可执行动作是什么。
- 用户完成后下一步怎么自然推进。
- 用户卡住、信息不足或状态不好时，如何降低阻力继续走。

## 演示前准备

1. 启动服务：

```bash
npm run dev
```

2. 打开前端：

```txt
http://localhost:5173
```

3. 准备终端验证命令：

```bash
npm run test:all
```

## 演示路径

### 1. 任务启动

输入一个具体任务：

```txt
写一版作品集项目介绍
```

讲解重点：

- 系统不会先生成一整套计划。
- 它会优先给出当前可以立刻开始的一步。
- StepCard 是整个产品的核心界面单元。

### 2. 连续推进

点击“开始执行”，再点击“完成当前步骤”。

讲解重点：

- 完成当前步骤后，系统会根据 `stepHistory` 推进下一步。
- 产品目标是从“生成器”变成“任务执行系统”。

### 3. 信息不足时先澄清

输入一个上下文不足的任务：

```txt
给我的店铺设计一个 Logo
```

讲解重点：

- 系统会先问店铺名、业务类型等必要信息。
- 澄清不是闲聊，而是为了避免 AI 在上下文不足时乱猜。

### 4. 用户卡住时降低阻力

在执行中点击“我卡住了”，输入：

```txt
我觉得写得太普通，拿不出手
```

讲解重点：

- 系统会识别完美主义阻力。
- fallback step 会降低质量门槛，例如先写 60 分版本。
- 这里展示的是任务内部阻力处理，不是心理分析。

### 5. 社交压力安全草稿

创建任务：

```txt
回老师消息
```

执行后卡住，输入：

```txt
我怕一打开就被老师说
```

讲解重点：

- 系统不应要求用户直接打开聊天框或发送消息。
- 正确策略是先写一个私密安全草稿。
- 这个场景有质量门禁自动测试。

## 质量证明

运行：

```bash
npm run test:all
```

讲解重点：

- `test:quality` 是硬性质量门禁，会在核心恢复策略退化时失败。
- `test:resistance` 覆盖多类卡点恢复样例。
- `build` 证明前端可以生产构建。

## 推荐收尾

可以用这句话收尾：

> 这个项目的价值不是“接入 AI”，而是把 AI 设计成一个能陪用户完成任务的执行闭环：它知道什么时候推进，什么时候澄清，什么时候降低阻力。
