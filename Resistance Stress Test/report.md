# Resistance System Stress Test Report

## 1. 测试目标

本报告基于 3 组 Resistance Stress Test Dataset，共 90 个高阻力样本，归纳当前 Resistance System 最容易失败的模式。

核心判断标准不是回复是否聪明，而是：

> 用户在点击“换一种方式继续”后，是否更可能重新进入执行。

因此，本报告重点关注：

- 系统是否识别真实阻力，而不是只看表面阻力
- fallback 是否能降低重新行动的门槛
- fallback 是否避免错误分类、模板安慰和无效推进
- fallback 是否避免让用户阻力更强

## 2. 总体结论

三组数据暴露出一个核心风险：

> 阻力系统如果只根据按钮类型决策，会把很多深层阻力误判成“步骤太大”或“普通拖延”。

最危险的失败不是答案不够丰富，而是：

- 用户害怕被责备，系统却继续让他发送消息
- 用户身体已经不适合执行，系统却继续拆动作
- 用户怀疑任务是否值得继续，系统却继续推进任务
- 用户卡在标准不清，系统却继续要求产出
- 用户被完美主义卡住，系统却让他再检查、再优化

这些 fallback 看似“推进”，但实际上会提高执行阻力，导致用户更难重新开始。

## 3. 高频失败模式

| failure_pattern | description | typical_surface_resistance | real_blocker | why_it_breaks_execution |
|---|---|---|---|---|
| 把情绪风险误判为普通拖延 | 用户不是不想做，而是害怕关系后果、责备、冲突或羞耻 | 不想做 | 被评价、被责备、关系破裂、冲突升级 | 继续拆步骤会逼近威胁源，让用户更想逃 |
| 把认知模糊误判为步骤过大 | 用户不是嫌步骤大，而是不知道判断标准、结构、方向或评价维度 | 太难了 | 标准缺失、结构缺失、任务边界不清 | 继续要求写一句、选一个，会让用户仍然不知道怎么做 |
| 把完美主义误判为质量不足 | 用户不是需要更多优化，而是无法接受粗糙版本被看见 | 不想做 / 太难了 | 暴露恐惧、专业形象焦虑、自我审查 | 让用户再检查、再润色，会强化完美主义循环 |
| 把身体状态阻力误判为执行意愿低 | 用户不是懒，而是困、累、疼、情绪崩溃或身体不适 | 当前状态不适合 | 生理能量不足、认知资源不足、身体不适 | 继续推进会让用户觉得系统不理解现实状态 |
| 把价值怀疑误判为执行卡顿 | 用户不是不会做下一步，而是不确定这件事还值不值得做 | 不确定是否继续 | 目标价值动摇、沉没成本、长期无反馈 | 继续推进任务会绕过真正问题，用户无法恢复行动意愿 |
| 把社交任务误判为事务处理 | 回消息、道歉、拒绝、催款表面是“发一条消息”，实际是情绪高风险动作 | 不想做 | 关系压力、冲突预期、讨好倾向 | 只提供话术或打开聊天框，不会解决发送前阻力 |
| 把低状态误判为可以微执行 | 很多低状态样本连“站起来”“打开文件”都可能过高 | 当前状态不适合 | 执行功能下降、身体疼痛、情绪耗竭 | 过早给动作会让用户失败感加重 |
| 把任务类型识别当作充分理解 | 即使识别出 writing / physical / learning，也可能忽略 currentStep 的具体卡点 | 太难了 / 不想做 | 卡点发生在当前步骤，不一定等同于任务类型 | fallback 会变成类型模板，而不是当前步骤解决方案 |

## 4. 五类场景的失败归纳

### A. 情绪阻力

主要样本：

- 回老师、导师、老板、客户、面试官消息
- 道歉、请假、拒绝、催款、解释迟到
- 面对家长、前任、亲戚、合作方

共同特征：

- 用户通常说“我不想点开”“我怕对方觉得我怎样”“我怕发出去更糟”
- 表面是“不想做”
- 真实根因是社交风险、被责备、关系破裂、内疚、羞耻或冲突预期

最容易失败的 fallback：

| bad_fallback_type | why_bad |
|---|---|
| 只让用户打开聊天框 | 打开聊天框正是威胁源，可能加剧回避 |
| 直接给发送文案 | 跳过情绪阻力，逼近高风险动作 |
| 鼓励“直接说清楚” | 忽略用户害怕后果，而不是不会表达 |
| 泛化安慰“别担心” | 没有行动入口，无法让用户重新执行 |

测试时应重点观察：

- fallback 是否先降低情绪威胁
- 是否避免直接要求发送
- 是否把目标从“完成沟通”降到“准备一个安全草稿 / 看一眼但不发送”
- 是否承认当前动作的社交风险

### B. 认知模糊

主要样本：

- 写论文、综述、开题、讨论部分
- 做作品集、商业计划书、产品面试、群面
- 找实习方向、论文选题、转专业申请

共同特征：

- 用户说“不知道写什么”“不知道标准是什么”“越看越乱”
- 表面是“太难了”
- 真实根因是标准缺失、结构缺失、评价维度缺失、任务边界过大

最容易失败的 fallback：

| bad_fallback_type | why_bad |
|---|---|
| 让用户先写一句 | 用户不知道写什么，一句也写不出来 |
| 让用户随便选一个 | 没有评价标准，选择仍然困难 |
| 继续读下一篇资料 | 增加输入量，可能让模糊更严重 |
| 给通用 2 分钟动作 | 动作小了，但认知入口仍不存在 |

测试时应重点观察：

- fallback 是否给出明确判断维度
- 是否把抽象任务变成可判断的小问题
- 是否减少输入量，而不是继续堆材料
- 是否让用户能从“不知道怎么做”转为“知道先判断什么”

### C. 完美主义

主要样本：

- 写简历、汇报稿、申请文书
- 做 PPT、作品集、设计稿、演讲稿
- 发布作品、录制视频、提交作业

共同特征：

- 用户说“太普通”“太丢脸”“还差一点”“怕别人觉得我菜”
- 表面是“不想做”或“太难了”
- 真实根因是暴露恐惧、自我审查、专业形象焦虑、无法容忍粗糙版本

最容易失败的 fallback：

| bad_fallback_type | why_bad |
|---|---|
| 让用户再检查一遍 | 强化完美主义检查循环 |
| 提供更多模板或选项 | 增加比较和选择负担 |
| 鼓励马上发布 / 发送 | 威胁过高，用户更可能逃避 |
| 继续优化标题 / 开场 | 把卡点继续放大 |

测试时应重点观察：

- fallback 是否允许低质量草稿存在
- 是否降低“被别人看到”的风险
- 是否避免“再优化”方向
- 是否把目标从“做得好”改成“留下一版可继续修改的东西”

### D. 身体状态阻力

主要样本：

- 健身、跑步、散步、打扫、整理
- 感冒、头疼、生理期、胃不舒服
- 加班后、困、情绪崩溃、脑子糊住

共同特征：

- 用户说“真的很累”“头疼”“困得要死”“一点小事都能哭”
- 表面是“当前状态不适合”
- 真实根因是身体能量不足、疼痛、睡眠不足、情绪调节资源不足

最容易失败的 fallback：

| bad_fallback_type | why_bad |
|---|---|
| 鼓励坚持 | 用户会觉得系统否认身体现实 |
| 继续拆成小动作 | 对低状态用户，小动作也可能过高 |
| 降低训练强度但仍推进 | 疼痛或不适时仍然可能有风险 |
| 把学习低效当成方法问题 | 实际是认知资源不足，不是方法不足 |

测试时应重点观察：

- fallback 是否允许正式暂停
- 是否保存中断点
- 是否避免把身体不适解释为懒
- 是否给出低负担重启入口，而不是继续推进当前动作

### E. 价值怀疑

主要样本：

- 是否继续考研、公考、留学、当前工作
- 是否继续创业项目、课题、副业、运营账号
- 是否继续写作、练琴、减肥、关系、长期项目

共同特征：

- 用户说“还值不值”“是不是骗自己”“是不是没希望”“是不是该停了”
- 表面是“不确定是否继续”
- 真实根因是目标价值动摇、长期无反馈、沉没成本、身份怀疑、路线有效性怀疑

最容易失败的 fallback：

| bad_fallback_type | why_bad |
|---|---|
| 继续拆下一步 | 用户怀疑的是方向，不是步骤 |
| 给励志鼓励 | 无法解决价值判断问题 |
| 直接给职业/人生建议 | 越界且不能促成用户自主判断 |
| 要求继续复盘数据 | 可能加重无意义感 |

测试时应重点观察：

- fallback 是否暂停执行推进
- 是否回到目标判断
- 是否帮助用户做最小价值确认
- 是否避免替用户做重大决定

## 5. 跨场景根因模式

| root_cause_cluster | appears_in | failure_risk |
|---|---|---|
| 害怕被评价 | 情绪阻力、完美主义、求职、作品发布 | 系统误以为是不会写或不想做 |
| 害怕关系后果 | 回消息、拒绝、道歉、催款、请假 | 系统误以为是普通事务 |
| 标准缺失 | 论文、作品集、面试、计划书 | 系统误以为是步骤太大 |
| 自我价值受损 | 求职、申请、作品、长期学习 | 系统误以为需要鼓励或模板 |
| 身体能量不足 | 健身、学习、打扫、出门 | 系统误以为可以靠微动作推进 |
| 长期无反馈 | 考研、运营、编程、练琴、减肥 | 系统误以为需要继续坚持 |
| 不可逆焦虑 | 提交、发送、发布、辞职、转向 | 系统误以为临门一脚需要催促 |

## 6. 表面阻力与真实阻力错配

| surface_resistance | common_real_root_causes | dangerous_wrong_response |
|---|---|---|
| 不想做 | 害怕被责备、羞耻、冲突、暴露、内疚 | 最小动作推进、直接打开入口 |
| 太难了 | 标准不清、结构不清、评价维度不清、完美主义 | 单纯拆小、继续产出 |
| 当前状态不适合 | 疼痛、困倦、能量耗尽、情绪崩溃 | 激励坚持、低强度继续做 |
| 不确定是否继续 | 价值动摇、沉没成本、方向怀疑、长期无反馈 | 继续推进下一步 |

关键结论：

> surface resistance 只能作为入口，不能作为最终决策依据。

真正决定 fallback 的应该是：

`task + currentStep + user_utterance + root_cause`

如果没有 user_utterance，系统至少要避免使用过度具体但可能错位的模板。

## 7. 最危险的误判类型排序

| rank | misclassification | severity | reason |
|---|---|---|---|
| 1 | 身体不适时继续推进 | 高 | 可能让用户感到被冒犯，甚至有实际风险 |
| 2 | 价值怀疑时继续执行 | 高 | 绕过根因，用户会觉得系统只会催任务 |
| 3 | 情绪风险时要求发送/联系 | 高 | 直接逼近威胁源，增加逃避 |
| 4 | 完美主义时要求再检查 | 中高 | 强化卡住循环 |
| 5 | 认知模糊时只拆小步骤 | 中高 | 动作变小但仍无法开始 |
| 6 | 泛化安慰但无动作 | 中 | 看起来友好，但不能恢复执行 |
| 7 | 类型模板覆盖 currentStep | 中 | 看似相关，但解决不了当前卡点 |

## 8. Fallback 有效性的判定标准

一个 fallback 是否通过测试，不看它是否“正确安慰”，而看它是否满足以下标准：

| criterion | pass_condition | fail_condition |
|---|---|---|
| 行动可进入 | 用户能在 10-60 秒内做出一个真实动作 | 仍然需要思考、选择、判断很久 |
| 根因对齐 | 解决真实阻力，而不是表面按钮类型 | 只根据“不想做/太难了”套模板 |
| 风险降低 | 降低社交、评价、身体或价值风险 | 逼近威胁源，要求发送、提交、公开 |
| 不增加负担 | 不引入更多选项、模板、检查项 | 给更多方案、更多标准、更多任务 |
| 保留推进感 | 即使暂停，也知道下次从哪里回来 | 只有安慰或完全中断 |
| currentStep 对齐 | 处理当前这一步的具体卡点 | 只按 task_type 给泛化动作 |

## 9. 适合后续自动化测试的检查点

| check | example_signal | expected_detection |
|---|---|---|
| 社交威胁词 | 怕他觉得、怕被骂、怕尴尬、被记恨、没人情味 | 情绪阻力优先于事务处理 |
| 完美主义词 | 太普通、丢脸、还差一点、不够高级、像废话 | 避免再检查/再优化 |
| 身体低状态词 | 头疼、困、胃不舒服、肚子疼、没电、撑不住 | 暂停/保存中断点优先 |
| 认知模糊词 | 不知道标准、不知道怎么分、不知道写什么、越看越乱 | 给判断维度而不是继续产出 |
| 价值怀疑词 | 值不值、没意义、是不是该停、骗自己、没希望 | 目标判断优先于执行推进 |
| 不可逆动作 | 发送、提交、发布、辞职、拒绝、催款 | 降低暴露风险，避免直接推进 |

## 10. 对当前测试集的产品含义

这 90 个样本说明，Resistance System 的核心难点不在于生成更像人的话，而在于：

> 在用户最容易逃离任务的瞬间，判断他到底是在逃避动作、逃避情绪、逃避判断，还是身体真的不适合继续。

如果系统只做“更小一步”，会在以下场景失败：

- 情绪阻力：更小一步仍然指向威胁源
- 认知模糊：更小一步仍然不知道怎么做
- 完美主义：更小一步仍然可能被评价
- 身体低状态：更小一步仍然超出资源
- 价值怀疑：更小一步绕过了是否值得的问题

最终高质量 fallback 应该让用户产生的不是“这句话好聪明”，而是：

> “这个我现在真的能做一下。”
